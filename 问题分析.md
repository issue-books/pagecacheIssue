# pagecacheIssue

**结论先行**：  
该 Pod 的 Java 进程（PID 7）存在**严重的 Undertow 临时文件句柄（FD）泄漏**。进程累计打开了 **84,443** 个文件描述符，其中 **82,320** 个与 Undertow 相关（占比 97% 以上）。这些泄漏的文件句柄在内核中积累了约 **17 GB** 的 Page Cache（文件页缓存），叠加 JVM 自身约 **6 GB** 的 RSS，导致容器整体内存使用（`memory.usage_in_bytes`）高达 **23 GB**，从而触发 Rancher 内存异常告警。

---

### 一、根因分析（基于你提供的数据）

| 指标 | 数值 | 说明 |
|------|------|------|
| `memory.usage_in_bytes` | ~23 GB | 容器当前总内存占用，异常偏高 |
| `rss` | ~5.95 GB | JVM 堆、元空间、线程、直接内存等实际使用内存 |
| `cache` / `active_file` | ~17 GB / ~15.3 GB | 绝大部分内存占用来自**文件页缓存（Page Cache）**，且处于活跃状态（active_file） |
| `total_rss_huge` | ~1.87 GB | JVM 使用了透明大页（THP），属于正常行为 |
| FD 总数 | 84,443 | 严重超标，正常 Web 进程通常在数百到数千 |
| Undertow FD | 82,320 | 绝大多数 FD 带有 "undertow" 关键字 |
| `fdinfo` 中 `pos` 偏移 | 累计约 45 GB，单 fd 平均 400–900 KB | 说明这些 FD 是**真实的磁盘文件**（非 Socket），且被大量读写 |

**关键推理：**

1. **不是内存泄漏，而是“句柄泄漏 + Page Cache 膨胀”**  
   如果是纯粹的 JVM Heap 泄漏，你会看到 `rss` 极高而 `cache` 很低。但此处 `cache`（17 GB）远大于 `rss`（6 GB），且 `active_file` 占了大头。这说明内核把大量磁盘文件内容缓存到了内存里。

2. **泄漏的是 Undertow 临时文件（极大概率是 Multipart 上传临时文件）**  
   Undertow 在处理文件上传（Multipart）时，会将超出内存阈值的数据写入 `java.io.tmpdir`（通常是 `/tmp` 下的 `undertow*.upload` 临时文件）。如果应用代码**没有正确关闭 `InputStream` / `OutputStream`**，或**没有在请求结束后清理临时文件**，这些句柄就会一直保留。由于进程仍持有 FD，即使文件被标记删除，其占用的磁盘数据块和 Page Cache 也无法被内核回收。

3. **资源限制过宽，导致问题被放大**  
   `memory.limit_in_bytes` 为无限制（`9223372036854771712`），`hierarchical_memory_limit` 高达约 250 GB。Pod 没有配置合理的内存上限，使得 8 万多个泄漏文件可以无限累积，直到吃满大量内存才被发现。

---

### 二、解决方案

#### 1. 应用代码层（根治）
重点排查**文件上传**和**大请求体处理**逻辑：

- **务必关闭流**：使用 `try-with-resources` 确保 `MultipartFile.getInputStream()`、`getOutputStream()`、`FileItem` 等在使用后被关闭。
  ```java
  try (InputStream is = file.getInputStream()) {
      // 处理上传
  } // 自动关闭
  ```
- **异常分支也要清理**：在 catch 块或最终 `finally` 块中显式删除临时文件（如 `fileItem.delete()` 或 `Files.deleteIfExists(tempPath)`）。如果请求被客户端中断或校验失败，Undertow 的临时文件默认不会自动清理。
- **避免自己持有句柄**：如果代码中手动引用了 Undertow 的 `FormData` 或临时文件对象，确保不阻塞 GC，并在请求结束时释放。

#### 2. 框架/中间件配置层
- **限制上传大小**：在 `application.yml` 中设置合理的 Multipart 上限，防止恶意/异常大请求生成过多临时文件。
  ```yaml
  spring:
    servlet:
      multipart:
        max-file-size: 10MB
        max-request-size: 50MB
        file-size-threshold: 0  # 超过即写磁盘，确保及时触发清理逻辑
  ```
- **指定并监控临时目录**：显式设置 JVM 参数 `-Djava.io.tmpdir=/app/tmp`，并确保该目录有磁盘容量和 inode 监控。不要依赖默认 `/tmp`。

#### 3. Kubernetes / 容器层（兜底与收敛）
- **设置合理的内存 Limit**：当前 Pod 几乎没有内存上限。应根据业务正常峰值设置 `resources.limits.memory`（例如 `8Gi` 或 `12Gi`），而不是 250 GB。
  ```yaml
  resources:
    limits:
      memory: "8Gi"
    requests:
      memory: "4Gi"
  ```
  **作用**：一旦泄漏导致内存超限，Pod 会被 OOMKilled 并重启，避免拖垮整台节点，同时第一时间暴露问题。
- **设置 JVM 内存参数**：显式指定 `-Xmx`、`-XX:MaxMetaspaceSize`、`-XX:MaxDirectMemorySize`，让 JVM RSS 有明确边界。

#### 4. 监控与应急
- **加告警**：在 Rancher/Prometheus 中增加两项告警：
    - 容器 `container_memory_working_set_bytes` 超过阈值（如 10 GB）。
    - 进程 FD 数量（`process_open_fds`）超过阈值（如 5000）。
- **应急止血**：立即重启 Pod 可以释放全部 FD 和 Page Cache，临时恢复内存。但**必须同步修复代码**，否则问题会复发。

---

### 三、总结

**一句话总结**：你的 Java Web 应用在使用 Undertow 时发生了**大规模文件上传临时文件句柄泄漏**（约 82,000 个），导致内核 Page Cache 膨胀至 17 GB，从而使 Pod 总内存达到 23 GB。

**解决优先级**：
1. **最高**：修复代码中文件流/临时文件的关闭与清理逻辑。
2. **高**：为 Pod 设置合理的 K8s Memory Limit，避免无限制膨胀。
3. **中**：完善上传大小限制与监控告警。
